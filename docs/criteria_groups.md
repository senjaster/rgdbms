# Группировка критериев тестирования СУБД

## Группа 1: Развертывание и установка
**1.** Возможность установки СУБД в закрытом контуре
**2.** Поддержка автоматизированной установки СУБД на операционных системах (РЕДОС, ALT linux, Astra Linux, SberLinux OS Server или др.) входящих в состав ЕРРПО
**3.** Поддержка автоматизированного обновления между минорными и мажорными версиями

## Группа 2: Сжатие данных
**6.** Возможность сжатия данных на уровне хранения

## Группа 3: Графическая консоль - базовый мониторинг и выполнение запросов
**10.** Наличие графической консоли мониторинга и управления
**19.** Возможность выполнения запросов к СУБД непосредственно из графической консоли
**20.** Возможность графического представления планов выполнения запросов
**21.** Возможность просмотра информации о структуре и размерах БД, объектах БД


## Группа 4: Графическая консоль - анализ исторических данных
**4.** Инструментарий для анализа производительности, с возможностью построения отчётов за различные промежутки времени, в том числе дифференциальных отчётов
**12.** Наличие преднастроенных метрик в графической консоли мониторинга и управления
**13.** Возможность добавления собственных метрик в графической консоли мониторинга и управления
**14.** Возможность просмотра исторических данных по объектам мониторинга через графическую консоль
**15.** Наличие агента для сбора информации о сервере СУБД


## Группа 5: Аутентификация и интеграция с внешними системами
**18.** Возможность интеграции графической консоли с внешней системой аутентификации
**25.** Возможность использования внешнего провайдера аутентификации
**27.** Настройка ролевой модели доступа на основе групп, атрибутов пользователя с использованием внешнего провайдера (LDAP)

## Группа 6: Шифрование
**24.** Возможность использования корпоративного TLS/SSL сертификата для компонентов решения
**35.** Наличие инструментов прозрачного шифрования данных на уровне хранения

## Группа 7: Парольные политики
**28.** Наличие встроенного механизма парольных политик для управления требованиями к сложности паролей (буквы, цифры, спецсимволы, запрет словарных слов, длина пароля)
**29.** Наличие встроенного механизма парольных политик для управления жизненным циклом паролей (запрет переиспользования, время жизни, блокировка по неуспешным входам или по времени)

## Группа 8: Аудит и безопасность
**32.** Наличие встроенного инструмента системного аудита действий пользователей, в том числе администраторов
**34.** Возможность хранения журнала аудита в выделенном файле

## Группа 9: Модели управления доступом
**36.** Поддержка дискреционной (DAC) модели управления доступом
**37.** Поддержка ролевой (RBAC) модели управления доступом

## Группа 10: Отказоустойчивость и автоматическое переключение
**44.** Наличие в поставке инструментов для построения отказоустойчивых конфигураций
**45.** Наличие в поставке инструментов, позволяющих автоматически переключаться на резервный узел, в случае сбоя в работе основного узла
**53.** Встроенный отказоустойчивый кластер, не требующий использования сторонних решений
**56.** Возможность проксирования запросов на активный узел (в случае переключения ролей)

## Группа 11: Резервное копирование и восстановление
**48.** Наличие в поставке инструментов, позволяющих выполнять резервное копирование и восстановление
**51.** Наличие в поставке инструментов, позволяющих выполнять резервное копирование и восстановление на S3
**52.** Восстановление из резервной копии отдельной БД

## Группа 12: Вертикальное масштабирование
**58.** Вертикально: параллельные планы выполнения
**59.** Вертикально: учет NUMA-нод при аллокации памяти

## Группа 13: Горизонтальное масштабирование - архитектура кластера
**57.** Поддержка конфигурации кластера с несколькими мастерами
**60.** Горизонтально: возможность построения active-active кластера
**61.** Поддержка многоузловых транзакций
**63.** Распределенное выполнение запроса: запрос может выполняться параллельно с использованием CPU на разных узлах
**55.** Наличие механизма балансировки читающей нагрузки
**64.** Наличие механизма автоматического распределения данных по узлам

## Группа 14: Горизонтальное масштабирование - эластичность
**67.** Compute-storage separation: наличие отдельных слоев, отвечающих за обработку запросов и за хранение данных
**68.** Наличие возможности расширять существующий кластер
**69.** Наличие возможности добавить узлы, отвечающие только за выполнение запросов
**70.** Эластичность: наличие механизмов, позволяющих динамически добавлять и исключать узлы из кластера в зависимости от нагрузки
**71.** Ограничения на практически возможный размер кластера

## Группа 15: Секционирование
**76.** Поддержка глобальных индексов для секционированных таблиц
**77.** Автосекционирование (автоматическое создание секций)

## Группа 16: Работа с Parquet
**80.** Возможность выгрузки данных штатными средствами из СУБД в parquet файлы
**81.** Возможность обращения штатными средствами СУБД к parquet файлам для выполнения аналитических запросов
**82.** Возможность использования в одном запросе данных, хранимых в СУБД и parquet файлах

## Группа 17: Очереди
**84.** Расширенное управление очередями (Advanced Queueing)

## Группа 18: Квотирование и приоритезация
**7.** Возможность квотирования или приоритезации ресурсов (cpu, ram, disk)
